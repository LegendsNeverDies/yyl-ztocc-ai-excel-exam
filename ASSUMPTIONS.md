# 需求理解与假设说明

> 本文档对应考点 6，覆盖题目故意留白的 9 项规则。每项给出具体数值/规则及其依据。
> 所有数值均实现为 `v3_config` / `v3_qc_rules` 可配置项，不硬编码，后台可调。

## ① 分级审批金额阈值

**设定**：金额 ≥ 500 元进入二级审批，< 500 元一级审批终审。

**依据**：
- 参考企业常见差旅/采购审批线（500-1000 元是一级与二级的分水岭）。
- 太低（如 100 元）会导致大量小额工单占用二级审批人带宽；太高（如 5000 元）则小额异常失去复核。
- 实现为 `v3_config.approval_threshold_l2 = 500`，配置中心可调，`approval-engine.ts` 的 `determineLevel()` 读取此值。

## ② 审批超时时长

**设定**：24 小时（1440 分钟）。

**依据**：
- 审批是人参与的环节，需覆盖跨天场景（晚上提交、次日上午处理）。
- 短于 4 小时会导致正常午休/夜间被误判超时；长于 48 小时则异常工单积压无法自动推进。
- 实现：`v3_config.approval_timeout_minutes = 1440`，`computeApprovalDue()` 计算 `dueAt`，列表/详情页基于 `dueAt` 显示超时角标，`triggerTimeoutCheck()` 懒触发自动升级。

## ③ 重提次数上限

**设定**：3 次。

**依据**：
- 给上报人修正机会（如补充描述、修正金额），但避免无限拉锯。
- 3 次后自动 `closed`，需重新发起上报（新工单号），避免同一工单长期占用审批资源。
- 实现：`v3_config.resubmit_limit = 3`，`rejectTicket()` 中 `resubmitCount > maxResubmit` 时转 `closed`。

## ④ 物流异常类型映射（各类型对应的下游动作）

| 异常类型 | 赔付方向 | 赔付类型 | 库存动作 | 说明 |
|---|---|---|---|---|
| 丢件 lost | 赔付客户 | claim | 回滚原运单库存 | 货已丢失，需赔付客户并回滚库存避免重复发货 |
| 破损 damaged | 赔付客户 | claim | 退货入库(+1) | 货物退回入库，赔付客户货损 |
| 客户拒收 rejected | 无赔付 | none | 退货入库(+1) | 仅退货入库 + 重新发货，不涉及赔付 |
| 超时未签收 timeout | 无赔付 | none | 无 | 仅重新发货，无赔付无库存变动 |
| 收货地址错误 address_error | 无赔付 | none | 无 | 更正地址重新发货，无赔付无库存变动 |

**依据**：基于物流行业常识——丢件/破损涉及实际货损需赔付，拒收/超时/地址错误多为流程问题，重新发货即可。映射在 `src/types/index.ts` 的 `EXCEPTION_ACTION_MAP`，`executeActions()` 据此联动。

## ⑤ 角色权限划分

| 角色 | 能上报 | 能扫描 | 一级审批 | 二级审批 | 误判快速放行 | 能转交 |
|---|---|---|---|---|---|---|
| operator 操作员 | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| approver1 一级审批人 | ❌ | ❌ | ✅ | ❌ | ❌ | ✅ |
| approver2 二级审批人 | ❌ | ❌ | ❌ | ✅ | ❌ | ✅ |
| qc_manager 品控主管 | ✅ | ✅ | ❌ | ❌ | ✅ | ❌ |

**关键约束**：
- **上报人不能审批自己提交的工单**（自批自核禁止）：`approveTicket()`/`rejectTicket()` 后端校验 `reported_by_id === user.userId` 则抛错，非仅前端隐藏。
- 角色可兼任：一个人理论上可拥有多个角色，但本系统简化为单角色（cookie `v3_current_user`），切换角色即换登录身份。
- 权限校验全部在 `src/lib/auth.ts` + server-actions，后端为准。

**单租户假设**：本系统 V2 数据不区分租户，V3 同样为单租户。若要支持多租户，需在 `v3_tickets`/`v3_users` 加 `tenant_id` 字段，所有查询加 `WHERE tenant_id = currentTenant`，运单快照表也需按租户隔离。当前实现未做多租户。

## ⑥ V2 数据同步频率与一致性策略

**同步模式**：实时拉取（realtime）为主，本地快照为降级兜底。

**具体策略**：
- **发起异常上报时**：必须实时调用 V2 `GET /api/external/waybills/:code` 校验运单存在，**不允许仅凭本地快照**（快照可能过期）。校验成功同时刷新本地快照。
- **扫描录入时**：实时调用 V2 `GET /api/external/waybills/:code/skus?skuCode=` 校验 SKU 归属。
- **工单详情页展示运单信息时**：实时拉取 V2，失败则降级到本地快照并明确标注"本地缓存，同步于 XX 时间"。
- **列表同步**：`listWaybills()` 提供批量拉取能力，可用于初始化或定期增量同步（本系统未做定时任务，按需手动触发）。

**V2 数据在异常处理期间变更的边界处理**：
- 采用**实时校验 + 差异记录**策略：每次关键动作（上报/审批）都重新拉取 V2 运单信息，若金额/收发件信息与工单创建时不一致，在审批记录中标注差异，由审批人人工判断。
- 不做自动对账（成本高），理由是异常处理周期通常较短（24h 内），V2 大规模变更概率低，人工兜底足够。
- 实现：`fetchWaybillByCode()` 每次调用都写 `v3_sync_logs`，可追溯任一时刻拉取的数据。

**依据**：实时校验成本可控（每次一个 HTTP 调用），但保证了"运单真实性"这一核心要求；定时批量同步会有窗口期，不适合"发起上报"这种关键动作。

## ⑦ 品控暂扣超时时长（独立于审批超时）

**设定**：2 小时（120 分钟），远短于审批超时（24 小时）。

**为何独立于审批超时**：
- 品控暂扣意味着**货物压仓**，占用仓库物理空间，产生实际运营成本（仓储费、压仓损耗、影响后续出库）。
- 审批超时是"流程卡顿"，成本是时间；品控暂扣是"实物积压"，成本是金钱 + 空间，两者性质不同，不能用同一阈值。
- 货物压仓 2 小时已是预警线（仓库通常按小时排产出），超过应强制升级处理。

**超时后处理**：强制升级进入二级审批（`nextStatusOnTimeout` 对品控工单返回 `level2_reviewing`），由二级审批人优先处理，避免货物长期积压。

**实现**：`v3_config.qc_hold_timeout_minutes = 120`（独立配置项），`computeQcHoldDue()` 计算，`triggerTimeoutCheck()` 触发升级。

## ⑧ 品控规则触发阈值

| 触发类型 | 阈值 | 严重度 | 自动审批层级 |
|---|---|---|---|
| 数量差异 quantity_diff | 差异 > 5% | high | 二级 |
| 破损等级 damage_level | 等级 ≥ 3（共 0-5 级） | medium | 二级 |
| 规格偏差 spec_deviation | 偏差 > 10% | medium | 二级 |
| 标签错误 label_error | 命中即触发 | low | 一级 |
| 批次异常 batch_anomaly | 命中即触发 | high | 二级 |

**依据**：
- 数量差异 5%：参考仓储盘点容差惯例（通常 1-3% 为正常损耗，> 5% 明显异常）。
- 破损等级 3 级：0=完好，1-2=轻微（可降级处理），3-5=严重（需追偿）。
- 规格偏差 10%：制造业常见公差阈值。
- 标签错误：低严重度（可重新贴标），但需记录追责供应商。
- 批次异常：高严重度（整批可能有问题），需追偿或退供应商。

**实现**：存于 `v3_qc_rules.trigger_condition` JSON 字段（如 `{"threshold_pct": 5}`），`qc-engine.ts` 的 `evaluateScan()` 读取并判定，命中后记录 `ruleId` + `reason` 到 `v3_scan_records`，可追溯。配置中心 UI 可新增/编辑/启停规则。

## ⑨ 品控主管角色权限边界

**能做什么**：
- 扫描录入（与操作员相同）
- 异常上报（与操作员相同）
- **误判快速放行**（独有）：对品控工单可绕过完整审批流程，直接解锁批次、关闭工单。**必须填写复核原因，留痕记录**（写入 `v3_approval_records` decision=`quick_release`），不允许静默放行。

**与审批人权限是否重叠**：
- 不重叠。品控主管**不能**做一级/二级审批（`canApproveLevel1`/`canApproveLevel2` 返回 false）。
- 一级/二级审批人**不能**做误判快速放行（`canQuickRelease` 返回 false）。
- 这是刻意设计：快速放行是"品控专业判断"权限，审批是"金额/流程"权限，分离避免权力集中。

**能否兼任**：本系统简化为单角色，但设计上 `canQuickRelease()` 只校验 `role === 'qc_manager'`，若未来允许一人多角色，需调整为权限集合判断。

**后端校验**：`quickRelease()` server-action 中 `if (!canQuickRelease(user)) throw new Error("仅品控主管可操作")`，前端隐藏按钮不算数。

## 主动澄清问题清单（如果能问产品经理）

以下是在需求不完整时，我会向产品经理提出的问题，体现主动澄清意识：

1. **分级审批金额阈值**：是否有财务制度规定的具体数值？还是由业务方自定？不同业务线（如生鲜 vs 日化）是否需要不同阈值？
2. **超时时长**：是否有 SLA 要求？夜间/节假日是否计入超时？还是 7×24 全计？
3. **重提次数**：拒绝后重新提交，金额能否调整？还是必须保持原金额？调整金额是否重置审批层级？
4. **异常类型映射**：丢件是否需要联动报警/保险？破损的退货入库是否需要质检复检？
5. **角色权限**：品控主管的"误判快速放行"是否有金额上限？大额品控异常是否也允许快速放行？
6. **V2 同步**：V2 运单数据变更频率多高？是否需要 V2 主动通知 V3（webhook）而非 V3 轮询？
7. **多租户**：未来是否会多租户？现在设计是否需要预留 tenant_id？
8. **AI 辅助**：AI 建议的展示位置和置信度阈值？AI 服务商是 DeepSeek 还是其他？
9. **批量场景**：200 条工单是演示规模，实际生产预期多少？是否需要分库分表？
10. **审计要求**：审批记录、赔付记录是否需要不可篡改（如区块链存证）？保留多久？

## 老系统二开意识（对应考点5）

### V2 新增对外接口的兼容性策略

V2 原本只有 `/api/ai/analyze` 和 `/api/rules/seed` 两个内部接口，没有对外接口。新增 `/api/external/waybills/*` 时采取以下策略：

1. **版本策略**：接口路径用 `/api/external/` 前缀独立命名空间，与 V2 既有 `/api/` 内部接口物理隔离。未来升级可用 `/api/external/v2/`，旧版本保持可用。
2. **字段向后兼容**：
   - V2 的 `shipments` 表字段（`externalCode`/`storeName`/`receiverName` 等）原样暴露，不改名不改类型。
   - 新增字段只能追加（如 `amount`），不删除既有字段。
   - V3 侧 `WaybillDetail` 类型对 V2 字段全部用 `| null` 兼容空值。
3. **灰度上线**：
   - 先在 V2 部署对外接口但不公开 URL，仅 V3 测试环境调用。
   - 验证通过后 V3 切换生产 URL，V2 接口正式启用。
   - V2 接口失败不阻塞 V2 既有功能（接口独立，不碰 V2 主流程）。
4. **不动 V2 主表**：异常标记回写用独立表 `v2_waybill_flags`（`CREATE TABLE IF NOT EXISTS`），不改 `shipments` 表结构，避免影响 V2 既有写入逻辑。

### V2 接口字段升级时 V3 的应对

**场景**：V2 的 `shipments.total_quantity` 从 `numeric` 改成 `decimal(12,2)`，或 `external_code` 改名。

**应对**：
1. **V3 不假设字段类型**：`v3_waybill_snapshots` 用自己的 `numeric(12,2)` 存储，V2 返回值 `String()` 转换后入库，类型变化不影响。
2. **V3 容错解析**：`fetchWaybillByCode()` 对 V2 返回 JSON 做字段存在性检查，缺失字段用 `null` 兜底，不抛错。
3. **接口契约文档**：`API_CONTRACT.md` 明确记录字段类型，V2 升级时对照检查，必要时 V3 侧加适配层。
4. **监控告警**：`v3_sync_logs` 记录每次调用的响应，若 V2 字段突变导致解析失败，日志会立即体现 `success=false`，可快速定位。

### 不破坏 V2 现有调用方的措施

- V2 的 `/api/ai/analyze`、`/api/rules/seed`、Server Actions 全部未动。
- 新增的 `/api/external/*` 是独立路由文件，不修改 V2 既有任何路由。
- V2 的 `next.config.ts` 未改（CORS 仍只对 `/api/*` 开放，`/api/external/*` 自动继承）。
- V2 的数据库表（`parse_rules`/`shipments`/`orders`）未改结构，仅新增 `v2_waybill_flags` 表（独立，IF NOT EXISTS 安全）。
